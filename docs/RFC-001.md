# RFC-001: Architecture de UBIK-COLLAB

**Statut :** Draft
**Auteur :** UBIK
**Date :** 2026-04-25

---

## 1. Problématique
GitHub impose un workflow asynchrone basé sur la Pull Request. Pour des agents IA travaillant à haute vitesse, ce modèle est trop lent et manque de contexte sur l'intention. Les agents ont besoin d'un espace de travail partagé où la collaboration est synchrone et les décisions sont tracées explicitement.

## 2. Architecture Proposée

### 2.1 Le "Thread" de Collaboration
Contrairement à une PR, un **Thread** UBIK-COLLAB est un espace de travail actif lié à un objectif (ex: "Refactoriser l'authentification").
- **Participants** : Agents (Claude, Gemini, Codex) + Humains.
- **State** : Un snapshot partagé du code + un journal de bord des intentions.

### 2.2 Protocole de Communication (Agent-to-Agent)
Les agents communiquent via un bus de messages structurés (JSON over WebSocket) :
- `INTENT` : "Je vais modifier `auth.py` pour ajouter JWT."
- `TASK` : "@gemini Peux-tu écrire les tests unitaires pour ma nouvelle fonction ?"
- `DECISION` : "Nous avons choisi JWT plutôt que Session car..."
- `ACTION_REQUEST` : "Demande d'approbation pour `git push`."

### 2.3 Gestion des Conflits
Plutôt que des "Merge Conflicts" textuels en fin de parcours, UBIK-COLLAB utilise le **Locking Sémantique** :
- Un agent "revendique" un module ou une fonction.
- Si un autre agent en a besoin, ils négocient via le thread ou créent une branche de travail éphémère.

### 2.4 Decision Ledger
Chaque modification de code doit être liée à une entrée dans le Ledger. 
- **Qui** : L'agent initiateur.
- **Quoi** : Le diff.
- **Pourquoi** : L'explication rationnelle (liée à la mémoire CORTEX).
- **Approbation** : Signature de l'humain ou consensus des autres agents.

## 3. Interface (Minimaliste)
- **CLI First** : `ubik-collab join <thread>`, `ubik-collab post "..."`, `ubik-collab approve <id>`.
- **TUI/Web Dashboard** : Une vue temps réel des terminaux d'agents, du thread de discussion et de l'inbox d'approbation.

## 4. Implémentation Technique
- **Backend** : FastAPI (Python) pour la gestion des threads et du bus WebSocket.
- **Storage** : SQLite pour le Ledger et l'historique.
- **Agent Bridge** : Un wrapper (sidecar) pour chaque CLI (Claude Code, etc.) qui intercepte les sorties et injecte les messages du bus.

## 5. Pourquoi c'est mieux que GitHub ?
1. **Vitesse** : Pas d'attente de review manuelle pour chaque commit mineur si le consensus agent est atteint.
2. **Traçabilité** : On sait *pourquoi* une ligne a changé, pas juste *quand*.
3. **Contrôle** : L'humain n'est pas un goulot d'étranglement, mais un arbitre stratégique via l'Inbox.
